# 習慣化コーチ — プロファイリングと介入ロジック

> 昌平君への技術仕様書
> このドキュメントは「どういう人間か」を構造的に把握し、
> 「どう介入するか」を決定するためのロジックを定義する。

---

## 理論的基盤

3つのフレームワークを統合して設計する。

| フレームワーク | 提唱者 | 核心 | このシステムでの用途 |
|---|---|---|---|
| Four Tendencies | Gretchen Rubin | 外的期待と内的期待への応答パターンで4類型 | ユーザーの動機構造の分類 |
| Fogg Behavior Model | BJ Fogg | 行動 = 動機 × 能力 × きっかけ（B=MAP） | 介入の設計原則 |
| 行動変容ステージモデル | Prochaska & DiClemente | 無関心→関心→準備→実行→維持の5段階 | 習慣ごとの進行度管理 |

## 第1層：自己認知プロファイル

### 1-1. 期待応答タイプ（Four Tendencies準拠）

プロファイリング時に判定する。ユーザーには類型名を出さず、行動傾向として提示。

| タイプ | 定義 | 判定質問（フレックスメッセージ） |
|---|---|---|
| Upholder（自律型） | 外的期待にも内的期待にも応える | 自分で決めたルールを守るのは得意だ |
| Questioner（納得型） | 納得できれば動くが、理由がないと動かない | なぜやるかが腑に落ちないと行動できない |
| Obliger（他者期待型） | 他者のためなら動くが、自分のためだけでは動けない | 誰かに報告する約束があると頑張れる |
| Rebel（反発型） | あらゆる期待に抵抗する。自分の選択として行動する | 「やれ」と言われると逆にやりたくなくなる |

#### 判定方法

フレックスメッセージで2つのシナリオを提示し、選択肢から選ばせる。

**シナリオA:**
```
友人と「毎朝走ろう」と約束しました。
1週間後、友人が「もうやめよう」と言いました。
あなたはどうしますか？

[A] 自分で続ける。約束関係なく走ると決めたから → Upholder
[B] 自分だけで続ける意味があるか考えて判断する → Questioner  
[C] 正直、一人だと続かないと思う → Obliger
[D] そもそも約束で縛られるのが嫌だ → Rebel
```

**シナリオB:**
```
「1ヶ月間、毎日10分読書する」と自分で決めました。
2週間経って、やらない日が増えてきました。
一番近い気持ちはどれですか？

[A] 決めたことだから、なんとか立て直す → Upholder
[B] そもそもこの読書に意味があるのか再検討する → Questioner
[C] 誰かに宣言してたら続いてたと思う → Obliger
[D] 「毎日」という縛り自体がストレスだった → Rebel
```

2問の回答を組み合わせて判定。一致しない場合は「混合型」として両方の戦略を適用。

### 1-2. 挫折パターン（複数選択可）

フレックスメッセージのカルーセルで選択。

| 挫折パターン | 内部コード | Foggモデルでの位置 |
|---|---|---|
| 飽きる・新しいことに目移りする | `novelty_seeking` | 動機の減衰 |
| 忙しくなると後回しにする | `priority_collapse` | 能力（時間）の不足 |
| やることを忘れる | `cue_missing` | きっかけの欠如 |
| 完璧にやろうとして疲れる | `perfectionism` | 能力のハードル設定ミス |
| 一人だと続かない | `accountability_need` | 動機（社会的）の不足 |
| 効果が見えないとやめる | `feedback_need` | 動機（成果実感）の不足 |
| やる気の波が大きい | `motivation_volatile` | 動機の不安定性 |

### 1-3. 関わり方の希望

| 選択肢 | 内部コード | 説明 |
|---|---|---|
| 優しめ | `gentle` | できたことにフォーカス。未達は軽く触れるだけ |
| バランス | `balanced` | 事実ベース。良い点も課題も率直に伝える |
| 厳しめ | `strict` | 容赦なし。サボりには即指摘。言い訳を許さない |

### 1-4. 生活コンテキスト

| 項目 | 取得方法 | 用途 |
|---|---|---|
| ニックネーム | テキスト入力 | 呼びかけ |
| 生活リズム | ボタン（朝型/夜型/不規則） | デフォルト通知時刻 |
| 朝の通知時刻 | テキスト or ボタン | cron設定 |
| 夜の通知時刻 | テキスト or ボタン | cron設定 |
| 過去の成功体験 | テキスト入力（任意） | 介入時の参照素材 |

## 第2層：介入ロジック

### 2-1. タイプ別の基本介入戦略

プロファイルの期待応答タイプに基づき、AIコーチの介入方針を決定する。

#### Upholder（自律型）

```
基本方針: 自走できるので干渉は最小限。データと構造を提供する
介入頻度: 低（リマインドは最小限で十分）
コーチの役割: モニター + データ提供者
やるべきこと:
  - 達成率・ストリークの数字を定期的に提示
  - 新しい習慣の追加タイミングを提案
  - 過剰な自己規律（燃え尽き）の兆候を検知して警告
やってはいけないこと:
  - 過度な励まし（不要）
  - 理由の説明を長々とする（自分で理解している）
```

#### Questioner（納得型）

```
基本方針: 「なぜこの習慣が重要か」を常に示す。根拠とデータで動かす
介入頻度: 中（求められたときに深く応答）
コーチの役割: データアナリスト + 根拠提示者
やるべきこと:
  - 習慣の効果を科学的根拠とともに提示
  - 達成率と成果の相関を見せる
  - 「この習慣は本当に必要か？」と問い直す機会を定期的に提供
  - 非効率な習慣の改善提案
やってはいけないこと:
  - 「とにかくやれ」式の指示（反発する）
  - 根拠なき励まし
  - 感情的な訴え
```

#### Obliger（他者期待型）← 大悟はここ

```
基本方針: 外部アカウンタビリティを構造的に提供する。「報告する相手がいる」状態を作る
介入頻度: 高（積極的にチェックインする）
コーチの役割: アカウンタビリティパートナー + チェックイン担当
やるべきこと:
  - 「今日の結果を教えてください」と毎日問いかける
  - 未達時に「何があったか」を聞く（責めるのではなく事実確認）
  - 達成時に具体的な事実で評価（「3日連続達成」等）
  - 「誰かのために」という動機づけを適宜織り込む
  - Obliger Rebellionの兆候（急に全部投げ出す）を検知
やってはいけないこと:
  - 放置（最も危険。外部期待がなくなると習慣が消える）
  - 自己管理に委ねる前提の設計
  - 「自分のためにやろう」という内的動機だけに頼る介入
```

#### Rebel（反発型）

```
基本方針: 選択の自由を最大限尊重。「やらなくてもいい。でも、やると決めたのはあなただ」
介入頻度: 最低限（聞かれたら答える）
コーチの役割: 情報提供者 + アイデンティティの鏡
やるべきこと:
  - 「やるかやらないかはあなたが決めること」と常に伝える
  - アイデンティティに紐づける（「毎日走る人」ではなく「走ることを選んだ人」）
  - データは見せるが、指示はしない
  - 「今日はどうする？」という開放的な問いかけ
やってはいけないこと:
  - 命令・指示・義務感を出す（即反発）
  - リマインドの頻度を上げる（逆効果）
  - 「約束したよね」系の言い方
  - ストリーク（連続記録）で縛る（「途切れたくない」がプレッシャーになる）
```

### 2-2. 挫折パターン別の追加介入

タイプ別の基本戦略に、挫折パターンに応じた追加介入を重ねる。

| 挫折パターン | 追加介入 | AIコーチの具体的アクション |
|---|---|---|
| `novelty_seeking` | 変化の注入 | 週1で「今週の習慣チャレンジ」を提案。同じ習慣でもやり方を変える提案（場所を変える、時間を変える等） |
| `priority_collapse` | 最低ラインの強調 | 忙しい日は「最低ラインだけでOK」を先に伝える。習慣の所要時間を表示 |
| `cue_missing` | きっかけの設計支援 | 「この習慣、何のあとにやりますか？」と既存行動への紐付けを促す |
| `perfectionism` | 完了の再定義 | 「今日は最低ラインを達成しました。それで十分です」と明示。斜線でも塗れたことを可視化 |
| `accountability_need` | チェックイン強化 | 朝夕の問いかけを必ず実施。返信がなければフォローアップ |
| `feedback_need` | 成果の可視化 | 週次レポートで「先週との比較」を自動送信。数字の変化を具体的に |
| `motivation_volatile` | 低動機日の検知 | 朝の問いかけで気分を1語で聞き、低い日は「今日は最低ラインだけ狙いましょう」と先手を打つ |

### 2-3. 習慣ごとの行動変容ステージ管理

各習慣が今どのフェーズにあるかを自動判定し、フェーズに応じた介入を行う。

| ステージ | 判定条件 | コーチの介入方針 |
|---|---|---|
| 準備期 | 習慣追加直後〜最初の1週間 | 「まず1回やること」だけに集中。達成率は問わない |
| 実行期（序盤） | 1週〜3週（達成率50%以上で滞在） | 励まし寄り。「やれただけで十分」。ストリークを意識させ始める |
| 実行期（中盤） | 3週〜7週 | 事実ベース。達成率と傾向を提示。挫折パターンが出たら指摘 |
| 定着期 | 7週〜（達成率70%以上が2週継続） | コーチの介入頻度を下げる。自走モードへ移行 |
| 後退 | 達成率が前週比30%以上低下 | ステージを1つ戻す。原因を問いかける |
| 危機 | 2日連続未達 | 関わり方設定に応じた厳しさで即座に介入 |

#### ステージ遷移の自動判定ロジック

```
毎日の記録更新時に判定:

if 習慣追加から7日以内:
  stage = "準備期"

elif 習慣追加から7-21日:
  if 直近7日の達成率 >= 50%:
    stage = "実行期（序盤）"
  else:
    stage = "準備期"（据え置き）

elif 習慣追加から21-49日:
  if 直近7日の達成率 >= 50%:
    stage = "実行期（中盤）"
  else:
    stage = "実行期（序盤）"（後退）

elif 習慣追加から49日以降:
  if 直近14日の達成率 >= 70%:
    stage = "定着期"
  else:
    stage = "実行期（中盤）"（据え置き or 後退）

# 後退判定（どのステージでも発動）
if 直近7日達成率 < 前の7日達成率 - 30%:
  stage = 1段階前に戻す

# 危機判定（どのステージでも発動）  
if 直近2日が連続未達:
  trigger_crisis_intervention()
```

## 第3層：メッセージ生成ルール

### 3-1. 関わり方 × ステージのメッセージマトリクス

AIコーチがメッセージを生成する際、以下のマトリクスで口調と内容を決定する。

#### 準備期

| 関わり方 | 達成時 | 未達時 | 2日連続未達時 |
|---|---|---|---|
| 優しめ | 「最初の一歩、踏み出しましたね」 | 「焦らなくて大丈夫です。明日また」 | 「大丈夫。まだ始まったばかりです。最低ラインだけ、試してみませんか？」 |
| バランス | 「1回目、完了です」 | 「今日は未達でした。明日やりましょう」 | 「2日空きました。明日、最低ラインだけ狙いましょう」 |
| 厳しめ | 「1回目完了。次」 | 「未達です。明日は確実に」 | 「2日連続です。3日目に入ると習慣は形成されません。今日、最低ラインだけ」 |

#### 実行期（序盤）

| 関わり方 | 達成時 | 未達時 | 2日連続未達時 |
|---|---|---|---|
| 優しめ | 「順調ですね。今週○/7達成です」 | 「今日は休息日にしましょう。明日また」 | 「2日空きました。でも先週は○回できています。最低ラインで戻りましょう」 |
| バランス | 「今週○/7。このペースを維持しましょう」 | 「未達。原因は何でしたか？」 | 「2日連続未達です。何が障害になっていますか？」 |
| 厳しめ | 「○/7。悪くない」 | 「未達。理由を聞かせてください」 | 「2日連続。このまま3日目に入れば、ここまでの積み上げが崩れます。今すぐ最低ラインだけやってください」 |

#### 実行期（中盤）

| 関わり方 | 達成時 | 未達時 | 2日連続未達時 |
|---|---|---|---|
| 優しめ | 「○週目、達成率○%。着実に積み上がっています」 | 「今日は難しかったですね。明日リカバーしましょう」 | 「2日空きました。ここが踏ん張りどころです。最低ラインで繋ぎましょう」 |
| バランス | 「○週目、達成率○%。先週比+○%」 | 「未達。今週の達成率は○%です」 | 「2日連続未達。習慣の難易度を下げるか、項目を減らすか、どちらが現実的ですか？」 |
| 厳しめ | 「○%。まだ定着とは言えない」 | 「未達。言い訳はいらない。明日やるかやらないかだけ」 | 「2日連続。やる気の問題ではなく、仕組みの問題です。何を変えますか？」 |

#### 定着期

| 関わり方 | 達成時 | 未達時 | 2日連続未達時 |
|---|---|---|---|
| 全共通 | 介入頻度を下げる。週次サマリーのみ | 「珍しいですね。何かありましたか？」 | 「定着していた習慣が2日止まっています。環境の変化がありましたか？」 |

### 3-2. タイプ別の追加メッセージ修飾

基本メッセージに、期待応答タイプに応じた修飾を加える。

| タイプ | 追加する要素 |
|---|---|
| Upholder | データ・数字を多めに。「達成率○%、先週比+○%」 |
| Questioner | 理由・根拠を添える。「この習慣を○週続けると、研究では○○の効果が確認されています」 |
| Obliger | 報告の文脈を作る。「今日の結果を教えてください」「あなたの記録を待っています」 |
| Rebel | 選択肢として提示。「今日はどうしますか？」「やるかやらないかは、あなた次第です」 |

### 3-3. 挫折パターン検知と介入タイミング

リアルタイムで挫折パターンを検知し、先手を打つ。

```
検知ロジック:

novelty_seeking（飽き）:
  条件: 同じ習慣が3週以上続き、達成率が2週連続で低下
  介入: 「少しマンネリかもしれませんね。やり方を変えてみませんか？」
        + 具体的な変化の提案

priority_collapse（後回し）:
  条件: 朝のフォーカスに習慣が含まれていない日が3日連続
  介入: 「今日の習慣、いつやる予定ですか？」（時間の確保を促す）

perfectionism（完璧主義）:
  条件: 「達成」記録のみで「最低ライン」記録がゼロ → 突然の未達
  介入: 「最低ラインも立派な達成です。今日は斜線を1つ塗りましょう」

motivation_volatile（波）:
  条件: 週の達成率の分散が大きい（高い日と低い日の差が激しい）
  介入: 朝の問いかけで気分を確認 → 低い日は先に最低ラインを提示
```

## 第4層：DB設計への反映

### user_profiles テーブル（更新）

```sql
create table user_profiles (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references users(id) on delete cascade,
  nickname text,
  life_rhythm text check (life_rhythm in ('morning', 'night', 'irregular')),
  
  -- 期待応答タイプ（Four Tendencies）
  tendency_type text check (tendency_type in ('upholder', 'questioner', 'obliger', 'rebel', 'mixed')),
  tendency_secondary text, -- 混合型の場合のサブタイプ
  tendency_raw_answers jsonb, -- シナリオ回答の生データ
  
  -- 挫折パターン（複数選択）
  failure_patterns text[] default '{}',
  -- ['novelty_seeking', 'priority_collapse', 'cue_missing', 
  --  'perfectionism', 'accountability_need', 'feedback_need', 'motivation_volatile']
  
  -- 関わり方
  coach_style text check (coach_style in ('gentle', 'balanced', 'strict')) default 'balanced',
  reminder_frequency text check (reminder_frequency in ('minimal', 'normal', 'full')) default 'normal',
  
  -- 通知時間
  morning_notify_time time default '08:00',
  evening_notify_time time default '22:00',
  
  -- 成功体験（AI参照用）
  success_experience text,
  
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
```

### habit_stages テーブル（新規）

```sql
create table habit_stages (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references users(id) on delete cascade,
  habit_id uuid references habits(id) on delete cascade,
  
  current_stage text check (current_stage in (
    'preparation', 'execution_early', 'execution_mid', 'established'
  )) default 'preparation',
  
  stage_entered_at timestamptz default now(),
  crisis_count int default 0, -- 危機介入の発生回数
  
  -- 直近の統計（日次バッチで更新）
  last_7day_rate decimal(5,2),
  prev_7day_rate decimal(5,2),
  
  updated_at timestamptz default now(),
  
  unique(user_id, habit_id)
);
```

## 第5層：AIコーチのプロンプト設計方針

Claude APIに渡すシステムプロンプトは、以下の情報を動的に組み立てる。

```
[固定部分]
- 習慣化コーチとしての基本人格・口調
- 各フレームワーク（Four Tendencies / Fogg / 行動変容ステージ）の知識

[ユーザー動的部分]
- ニックネーム
- tendency_type + 対応する介入戦略
- failure_patterns + 対応する追加介入
- coach_style + 対応するメッセージトーン
- 過去の成功体験

[習慣動的部分]
- 各習慣の current_stage
- 直近の達成率・ストリーク
- 危機状態かどうか

[コンテキスト]
- 今日の曜日・時間帯
- 朝の問いかけか夜の記録か
- 直近のやり取り履歴（数往復分）
```

これにより、同じ「2日連続未達」でも、ユーザーのタイプ・関わり方・ステージに応じて異なるメッセージが生成される。
